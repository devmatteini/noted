# Android Project Structure

This explains the files Android Studio generated for this project. Analogies use Node.js and Rust
where helpful.

## Root Files

### `settings.gradle.kts`

Defines the Gradle project itself.

Purpose:

- names the root project: `Noted`
- includes modules: currently only `:app`
- configures where Gradle can download plugins and dependencies from

Analogy:

- close to a workspace/root config
- Rust: root `Cargo.toml` with `[workspace]` members
- Node: monorepo workspace config listing packages

This file does not usually contain app dependencies. It says which modules exist and which
repositories Gradle may use.

### `build.gradle.kts`

Root Gradle build file.

Purpose:

- declares Gradle plugins available to subprojects
- uses `apply false` so plugins are known but not applied at the root

Analogy:

- Node: root-level shared build tooling config
- Rust: workspace-level dependency/tooling setup

For this project, the real Android app config lives in `app/build.gradle.kts`.

### `gradle.properties`

Global Gradle settings for this project.

Purpose:

- configures Gradle JVM memory
- enables Gradle configuration cache
- sets Kotlin code style

Analogy:

- Node: `.npmrc` or tool-specific config
- Rust: `.cargo/config.toml`, but for Gradle behavior

### `gradlew`

Unix shell script for the Gradle Wrapper.

Purpose:

- runs the project's pinned Gradle version without requiring Gradle installed globally
- downloads Gradle if needed

Use this instead of system `gradle`:

```sh
./gradlew :app:assembleDebug
```

Analogy:

- Node: `corepack`/pinned package manager behavior
- Rust: less common, but similar spirit to using a toolchain-pinned command instead of random global
  tooling

### `gradlew.bat`

Windows version of `gradlew`.

Purpose:

- same as `gradlew`, but for Windows shells

Keep it committed so Windows users/CI can build the project.

### `local.properties`

Local machine config generated by Android Studio.

Purpose:

- usually contains `sdk.dir=/path/to/Android/Sdk`
- tells Gradle where your Android SDK is

Do not commit this file. It is machine-specific.

Analogy:

- Node: `.env.local`
- Rust: local-only tool path config

## Gradle Directory

### `gradle/libs.versions.toml`

Gradle version catalog.

Purpose:

- centralizes dependency versions
- centralizes plugin versions
- gives names like `libs.androidx.compose.ui` used in Gradle files

Analogy:

- closest Android/Gradle equivalent to `package.json` dependencies
- Rust: `[dependencies]` versions in `Cargo.toml`, but centralized for all modules

Example from this project:

```toml
androidx-compose-ui = { group = "androidx.compose.ui", name = "ui" }
android-application = { id = "com.android.application", version.ref = "agp" }
```

Then `app/build.gradle.kts` can use:

```kotlin
implementation(libs.androidx.compose.ui)
alias(libs.plugins.android.application)
```

### `gradle/wrapper/gradle-wrapper.properties`

Gradle Wrapper config.

Purpose:

- pins the Gradle version
- points to the Gradle distribution URL

Current project uses Gradle `9.4.1`.

Analogy:

- Node: `packageManager` field in `package.json`, or committed lock around tool version
- Rust: `rust-toolchain.toml` for toolchain pinning

### `gradle/wrapper/gradle-wrapper.jar`

Small binary used by `gradlew`.

Purpose:

- bootstraps/downloads/runs the pinned Gradle distribution

This should be committed. Without it, `./gradlew` may not work on a clean checkout.

### `gradle/gradle-daemon-jvm.properties`

Generated Gradle daemon JVM config.

Purpose:

- helps Gradle choose/manage a JVM for the daemon
- Android Studio may generate this for newer Gradle/JDK setups

Analogy:

- Node: local tool runtime config
- Rust: toolchain/runtime selection config

## App Module

### `app/`

The Android application module.

Purpose:

- contains app source code
- contains Android resources
- contains app-specific Gradle config
- produces the APK

Current root project has one module: `:app`.

### `app/build.gradle.kts`

Main app build config.

Purpose:

- applies Android and Kotlin/Compose plugins
- sets package namespace/application ID
- sets compile SDK, target SDK, min SDK
- configures build types
- enables Compose
- declares app dependencies

Closest analogy:

- Node: `package.json` for the app package
- Rust: `Cargo.toml` for the app crate

Important fields:

```kotlin
namespace = "com.cosimomatteini.noted"
applicationId = "com.cosimomatteini.noted"
minSdk = 35
targetSdk = 36
```

Notes:

- `namespace` is the code/resource namespace used by Android tooling.
- `applicationId` is the installed app ID on Android devices.
- They are often the same, but they are not exactly the same concept.

### `app/build/`

Generated build output.

Purpose:

- compiled classes
- generated resources
- APKs
- compiler caches

Do not commit this directory.

Analogy:

- Node: `dist/`, `.next/`, `build/`
- Rust: `target/`

## App Source Sets

Android modules organize code by source set.

### `app/src/main/`

Main production app source set.

Purpose:

- production Kotlin code
- Android manifest
- app resources

### `app/src/test/`

Local JVM tests.

Purpose:

- tests that run on your development machine
- no emulator/device needed
- good for domain logic, pure Kotlin, validation rules

Current file:

- `app/src/test/java/com/cosimomatteini/noted/ExampleUnitTest.kt`

### `app/src/androidTest/`

Instrumented Android tests.

Purpose:

- tests that run on an Android device or emulator
- can access Android framework APIs
- used for UI tests, database tests if Android context is needed, integration behavior

Analogy:

- Node: browser/e2e tests running in a real browser instead of plain Node
- Rust: integration tests requiring an external runtime/service

Current file:

- `app/src/androidTest/java/com/cosimomatteini/noted/ExampleInstrumentedTest.kt`

## Android Manifest

### `app/src/main/AndroidManifest.xml`

The app manifest.

Purpose:

- declares app components
- declares launcher activity
- declares permissions later
- declares app label/icon/theme
- configures backup behavior

Analogy:

- Node: partly `package.json` metadata plus runtime entrypoint config
- Rust: no exact equivalent; closest is crate metadata plus OS/service manifest

Important current section:

```xml

<activity android:name=".MainActivity" android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
```

This makes `MainActivity` the app launcher entry point.

## Kotlin Source

### `app/src/main/java/`

Production Kotlin/Java package root.

Purpose:

- contains app code organized by package path

Even Kotlin projects often use a directory named `java` because Android/Gradle historically used
that source root for both Java and Kotlin.

### `app/src/main/java/com/cosimomatteini/noted/`

Package directory for `com.cosimomatteini.noted`.

Purpose:

- mirrors the Kotlin package declaration
- contains app code under this namespace

### `MainActivity.kt`

Current Android entry point.

Purpose:

- defines `MainActivity`, the launcher activity declared in the manifest
- calls `setContent { ... }` to render Compose UI
- currently shows a generated `Greeting` composable

Analogy:

- Node frontend: `main.tsx`, `App.tsx`, or route entry where React mounts
- Rust GUI/server: `main.rs` entry point, though Android lifecycle owns construction

Key concepts:

- `ComponentActivity` is an Android screen/activity base class.
- `onCreate` is lifecycle callback called when the activity starts.
- `setContent` mounts Compose UI.
- `@Composable` functions describe UI.
- `@Preview` lets Android Studio render the composable preview.

## Android Resources

### `app/src/main/res/`

Android resource directory.

Purpose:

- XML resources
- strings
- icons
- theme/style resources
- images/drawables

Resources are compiled by Android tooling and referenced as `R.string.app_name`,
`R.mipmap.ic_launcher`, etc.

Analogy:

- Node frontend: `public/` plus structured assets and design XML
- Rust: embedded/compiled assets, but Android has first-class resource compilation

### `app/src/main/res/values/`

Simple XML value resources.

Purpose:

- strings
- colors
- themes/styles

Files here do not represent runtime code. They are compiled into Android resources.

### `app/src/main/res/values/strings.xml`

String resources.

Purpose:

- contains app-visible strings like app name
- supports localization later by adding locale-specific folders

Current value:

```xml

<string name="app_name">Noted</string>
```

Analogy:

- frontend i18n message file

### `app/src/main/res/values/themes.xml`

Android platform theme resource.

Purpose:

- defines `Theme.Noted`
- used by manifest before Compose starts
- controls base Android window/activity styling

This is separate from Compose `NotedTheme` in `Theme.kt`.

Rule of thumb:

- `themes.xml` affects the Android window and platform theme.
- `Theme.kt` affects Compose UI inside the window.

### `app/src/main/res/values/colors.xml`

XML color resources.

Purpose:

- legacy/template Android color values
- useful for XML drawables/themes

This is separate from Compose colors in `Color.kt`.

### `app/src/main/res/drawable/`

Drawable resources.

Purpose:

- vector images
- shape XML
- bitmap-like drawable definitions
- generated launcher foreground/background assets

Current files include launcher icon drawable parts.

Analogy:

- frontend SVG/image assets

### `app/src/main/res/mipmap-*dpi/`

Launcher icon assets for different screen densities.

Purpose:

- Android chooses the best icon resolution for a device
- `mipmap-hdpi`, `mipmap-xhdpi`, etc. are density buckets

Use `mipmap` for launcher icons. Use `drawable` for normal in-app images.

### `app/src/main/res/mipmap-anydpi/`

Adaptive launcher icon XML.

Purpose:

- defines adaptive icon layers usable at any density
- references foreground/background drawables

### `app/src/main/res/xml/backup_rules.xml`

Backup rules for Android backup.

Purpose:

- controls what app data Android may back up/restore

Usually template-generated unless the app has sensitive/local-only data.

### `app/src/main/res/xml/data_extraction_rules.xml`

Android 12+ data extraction config.

Purpose:

- controls cloud backup and device-transfer behavior

Also usually template-generated at project start.

## Mental Model

Android project layers:

1. Gradle root config: `settings.gradle.kts`, `build.gradle.kts`, `gradle.properties`.
2. Dependency catalog: `gradle/libs.versions.toml`.
3. App module config: `app/build.gradle.kts`.
4. Android declaration: `AndroidManifest.xml`.
5. Kotlin app code: `app/src/main/java/...`.
6. Android resources: `app/src/main/res/...`.
7. Tests: `app/src/test` and `app/src/androidTest`.
8. Generated output/cache: `.gradle/`, `build/`, `app/build/`.

Closest common-file analogies:

- `app/build.gradle.kts` ~= app `package.json` or crate `Cargo.toml`.
- `gradle/libs.versions.toml` ~= centralized dependency version list.
- `settings.gradle.kts` ~= workspace member list.
- `AndroidManifest.xml` ~= app runtime manifest/entrypoint declaration.
- `MainActivity.kt` ~= UI entry point like `main.tsx`/`App.tsx`.
- `res/` ~= compiled assets/resources.
- `gradlew` + `gradle-wrapper.jar` ~= pinned build tool bootstrap.
